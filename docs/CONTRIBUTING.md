Данный документ описывает правила и порядок участия в разработке проекта **ParkStaff**.

- исправить ошибку;
- предложить улучшение;
- добавить новую функциональность;
- обновить документацию.

## 1. Общие принципы

При участии в разработке рекомендуется придерживаться следующих принципов:

- не вносить изменения напрямую в ветку `main`;
- каждое изменение оформлять через отдельную ветку и Pull Request;
- описывать все изменения в коммитах и PR;
- не ломать существующую функциональность;
- следить за качеством кода и его читаемостью;
- обновлять документацию при изменении логики работы.

## 2. Подготовка окружения

### Клонирование репозитория

```bash
git clone https://github.com/AlBarsStudio/park-staff.git
cd park-staff
Установка зависимостей
Bash

npm install
Настройка переменных окружения
Создайте файл .env в корне проекта:

env

VITE_SUPABASE_URL=your_supabase_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_key
Запуск в режиме разработки
Bash

npm run dev
3. Порядок внесения изменений
Шаг 1: Создание ветки
Создайте ветку от main с понятным названием:

Bash

git checkout main
git pull origin main
git checkout -b feature/название-функции
Шаг 2: Внесение изменений
Внесите необходимые изменения в код или документацию.

Шаг 3: Проверка
Убедитесь, что:

проект корректно собирается: npm run build;
нет ошибок в консоли;
изменения не нарушают работу существующих функций.
Шаг 4: Коммит
Оформите коммит в соответствии с правилами (см. ниже).

Шаг 5: Push
Bash

git push origin feature/название-функции
Шаг 6: Pull Request
Создайте Pull Request в ветку main через GitHub.

4. Правила оформления коммитов
Формат сообщения коммита
text

<тип>: <краткое описание>
Типы коммитов
Тип	Описание
feat	Новая функциональность
fix	Исправление ошибки
docs	Изменение документации
style	Изменение стилей или форматирования кода
refactor	Рефакторинг кода без изменения функциональности
perf	Улучшение производительности
test	Добавление или изменение тестов
chore	Обновление зависимостей, конфигурации и прочее
Примеры
text

feat: добавлен экспорт графика в CSV
fix: исправлена ошибка удаления смены на текущий день
docs: обновлена инструкция для администратора
style: выравнивание отступов в компоненте ScheduleView
refactor: вынесена логика валидации дат в отдельную функцию
chore: обновлены зависимости React и Vite
Язык
Коммиты оформляются на русском языке.

5. Правила оформления Pull Request
Название PR
Название должно кратко описывать суть изменений:

что было проверено;

6. Работа с ветками
Основные ветки
Ветка	Назначение
main	Основная ветка, содержит production-код
Именование рабочих веток
Тип	Формат	Пример
Новая функция	feature/<название>	feature/export-schedule
Исправление	fix/<название>	fix/shift-deletion-bug
Документация	docs/<название>	docs/update-admin-guide
Рефакторинг	refactor/<название>	refactor/database-service
Правила
рабочие ветки создаются от main;
после слияния PR рабочая ветка удаляется;
прямые пуши в main не допускаются.
7. Стиль кода
TypeScript / React
используйте TypeScript для всех новых файлов;
следуйте существующему стилю проекта;
используйте функциональные компоненты;
выносите типы в отдельные интерфейсы;
используйте осмысленные имена переменных и функций;
избегайте any без крайней необходимости.
CSS / Tailwind
используйте CSS-переменные для цветов: var(--primary), var(--text) и т.д.;
избегайте жёстко заданных цветов;
используйте утилитарные классы Tailwind;
обеспечивайте адаптивность для мобильных устройств.
Python (бот)
следуйте стилю PEP 8;
используйте docstring для функций;
обрабатывайте исключения;
логируйте ключевые операции.
Общие правила
не оставляйте закомментированный код без пояснения;
не используйте console.log в production-коде без необходимости;
удаляйте неиспользуемые импорты;
следите за единообразием форматирования.
8. Работа с документацией
Когда обновлять документацию
Документация обновляется, если:

добавлена новая функция;
изменена существующая логика;
изменена структура базы данных;
добавлены или изменены команды бота;
изменён процесс развёртывания.
Где находится документация
text

docs/
├── README.md           — навигация
├── user-guide.md       — инструкция для сотрудника
├── admin-guide.md      — инструкция для администратора
├── vk-bot.md           — руководство по боту
├── database.md         — структура базы данных
└── deployment.md       — деплой и настройка

CHANGELOG.md            — история изменений
CONTRIBUTING.md         — правила участия в разработке
Стиль документации
документация пишется на русском языке;
технические термины и названия сохраняются на английском;
используется формат Markdown;
каждый документ содержит оглавление;
каждый документ содержит раздел «Примечание» в конце.
9. Сообщение об ошибках

ожидаемое поведение;
фактическое поведение;
скриншоты (при необходимости);
браузер и устройство (если ошибка в интерфейсе).
