## 2. Локальная разработка

### 2.1. Клонирование репозитория

```bash
git clone https://github.com/AlBarsStudio/park-staff.git
cd park-staff
```

### 2.2. Установка зависимостей

```bash
npm install
```

### 2.3. Настройка переменных окружения

Создайте файл `.env` в корне проекта:

```env
VITE_SUPABASE_URL=https://your-project-id.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key
```

Значения берутся из панели управления Supabase.

Подробнее: [раздел «Получение ключей доступа»](#32-получение-ключей-доступа).

### 2.4. Запуск проекта

```bash
npm run dev
```

После запуска сайт доступен по адресу:

```
http://localhost:5173
```

### 2.5. Сборка проекта

Для создания production-сборки:

```bash
npm run build
```

Результат сборки будет в папке `dist/`.

Для локального предпросмотра собранной версии:

```bash
npm run preview
```

## 3. Настройка Supabase

### 3.1. Создание проекта

1. Перейдите на сайт: https://supabase.com
2. Создайте аккаунт или войдите.
3. Нажмите **New Project**.
4. Укажите название проекта и пароль для базы данных.
5. Выберите регион размещения.
6. Дождитесь завершения создания проекта.

### 3.2. Получение ключей доступа

1. Откройте проект в панели управления Supabase.
2. Перейдите в **Project Settings → API**.
3. Скопируйте:
   - **Project URL** — это значение для `VITE_SUPABASE_URL`;
   - **anon public** — это значение для `VITE_SUPABASE_ANON_KEY`.

### 3.3. Создание таблиц

Перейдите в **SQL Editor** в панели управления Supabase и выполните SQL-запросы для создания таблиц.

Полное описание структуры таблиц приведено в документе:

- [Документация по базе данных](database.md)

### 3.4. Настройка аутентификации

1. Перейдите в **Authentication → Settings**.
2. Убедитесь, что включена аутентификация по **Email**.
3. При необходимости для тестовой среды можно отключить подтверждение email.

### 3.5. Добавление пользователей

Добавление пользователей происходит в два этапа.

#### Шаг 1: Создание учётной записи в Supabase Auth

1. Перейдите в **Authentication → Users**.
2. Нажмите **Add User**.
3. Укажите email и пароль.
4. Скопируйте UUID созданного пользователя.

#### Шаг 2: Создание записи в таблице

Для администратора:

```sql
INSERT INTO admins (full_name, auth_uid, access_level)
VALUES ('Иванов Иван Иванович', 'uuid-из-auth', 2);
```

Для сотрудника:

```sql
INSERT INTO employees (full_name, auth_uid, access_level, base_hourly_rate)
VALUES ('Петров Пётр Петрович', 'uuid-из-auth', 3, 250);
```

#### Значения access_level

| Значение | Роль |
|---|---|
| `1` | Супер-администратор |
| `2` | Администратор |
| `3` | Сотрудник |

### 3.6. Row Level Security

Рекомендуется настроить политики RLS для защиты данных.

#### Базовые принципы

- сотрудник должен видеть и изменять только свои записи;
- администратор должен иметь доступ ко всем данным;
- анонимный доступ к данным должен быть закрыт.

#### Пример политики для таблицы `employee_availability`

```sql
-- Включить RLS
ALTER TABLE employee_availability ENABLE ROW LEVEL SECURITY;

-- Сотрудник видит только свои записи
CREATE POLICY "employee_own_availability" ON employee_availability
  FOR ALL
  USING (
    employee_id IN (
      SELECT id FROM employees WHERE auth_uid = auth.uid()
    )
  );

-- Администратор видит все записи
CREATE POLICY "admin_all_availability" ON employee_availability
  FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM admins WHERE auth_uid = auth.uid()
    )
  );
```

### 3.7. Realtime

Для работы подписок на изменения в реальном времени необходимо включить Realtime для нужных таблиц.

1. Перейдите в **Database → Replication**.
2. Включите Realtime для таблиц:
   - `employee_availability`
   - `schedule_assignments`
   - `actual_work_log`
   - `employee_attraction_priorities`
   - `employee_study_goals`

## 4. Деплой на GitHub Pages

### 4.1. Автоматический деплой

Проект настроен на автоматический деплой через **GitHub Actions**.

При каждом пуше в ветку `main`:

1. GitHub Actions запускает workflow из `.github/workflows/deploy.yml`.
2. Устанавливаются зависимости.
3. Выполняется сборка проекта.
4. Содержимое папки `dist/` публикуется на GitHub Pages.

### 4.2. Настройка секретов GitHub

Для корректной сборки необходимо добавить переменные окружения в секреты репозитория.

1. Перейдите в **Settings → Secrets and variables → Actions**.
2. Добавьте следующие секреты:

| Секрет | Значение |
|---|---|
| `VITE_SUPABASE_URL` | URL проекта Supabase |
| `VITE_SUPABASE_ANON_KEY` | Публичный anon-ключ Supabase |

### 4.3. Ручной деплой

При необходимости можно выполнить деплой вручную:

```bash
npm run build
npx gh-pages -d dist
```

## 5. Развёртывание бота ВКонтакте

### 5.1. Требования для бота

Для работы бота необходимо:

- сообщество ВКонтакте с включённым Callback API;
- токен доступа сообщества с правами на сообщения;
- серверная функция или сервер для обработки запросов;
- доступ к базе данных Supabase через Connection Pooler.

### 5.2. Настройка VK Callback API

1. Перейдите в **Управление сообществом → Работа с API → Callback API**.
2. Укажите URL вашей серверной функции.
3. Введите строку подтверждения (в коде бота: переменная `CONFIRMATION_CODE`).
4. В разделе **Типы событий** включите:
   - Входящее сообщение (`message_new`).
5. Сохраните настройки.

### 5.3. Настройка подключения к базе данных

Бот подключается к базе данных через **Connection Pooler** Supabase.

Параметры подключения:

| Параметр | Значение |
|---|---|
| `host` | `aws-0-<region>.pooler.supabase.com` |
| `port` | `6543` |
| `database` | `postgres` |
| `user` | `postgres.<project-ref>` |
| `password` | Пароль базы данных |
| `sslmode` | `require` |

Значения берутся из панели Supabase:

**Project Settings → Database → Connection string → Transaction pooler**

### 5.4. Маппинг сотрудников

Бот идентифицирует сотрудников по VK ID через маппинг в коде.

Формат маппинга:

```python
VK_TO_EMPLOYEE_MAP = {
    '<vk_user_id или screen_name>': <employee_id>,
    '123456789': 1001,
    'ivan_ivanov': 1002,
}
```

#### Как добавить нового сотрудника в маппинг

1. Узнайте VK ID или screen name сотрудника.
2. Узнайте его `id` в таблице `employees`.
3. Добавьте запись в словарь `VK_TO_EMPLOYEE_MAP`.
4. Задеплойте обновлённый код бота.

#### Как узнать VK ID

- Если у пользователя числовой ID, он виден в адресной строке профиля.
- Если у пользователя screen name, можно использовать его напрямую.
- Также можно использовать метод `users.get` VK API.

### 5.5. Деплой бота

Бот реализован как serverless-функция с точкой входа `handler(event, context)`.

#### Варианты размещения

| Платформа | Особенности |
|---|---|
| Yandex Cloud Functions | Нативная поддержка Python, удобная интеграция |
| AWS Lambda | Широкие возможности, требуется настройка API Gateway |
| VPS / Dedicated Server | Полный контроль, требуется настройка веб-сервера |

#### Порядок деплоя (общий)

1. Подготовьте файл бота и все зависимости.
2. Загрузите код на выбранную платформу.
3. Укажите точку входа: `handler`.
4. Настройте переменные окружения или вставьте конфигурацию в код.
5. Получите публичный URL функции.
6. Укажите этот URL в настройках Callback API сообщества ВКонтакте.
7. Выполните подтверждение сервера в настройках ВКонтакте.

## 8. Структура CI/CD

Файл: `.github/workflows/deploy.yml`

```text

       │
       ▼

       │
       ▼
Установка Node.js и зависимостей
       │
```

## 9. Обновление проекта

### Обновление сайта

1. Внесите изменения в код.
2. Выполните коммит и пуш в ветку `main`.
3. GitHub Actions автоматически соберёт и задеплоит обновление.
4. Проверьте результат на сайте.

### Обновление бота

1. Внесите изменения в код бота.
2. Задеплойте обновлённый код на serverless-платформу.
3. Проверьте работу бота, отправив команду `/ps тест` в беседу.

### Обновление базы данных

1. Подготовьте SQL-миграцию.
2. Выполните её через **SQL Editor** в панели Supabase.
3. Обновите документацию по базе данных.
4. Убедитесь, что сайт и бот корректно работают с новой структурой.

## 10. Диагностика проблем

### Сайт не загружается

| Симптом | Возможная причина | Решение |
|---|---|---|
| Белый экран | Ошибка сборки | Проверить консоль браузера |
| Ошибка подключения к Supabase | Неверные ключи | Проверить `.env` или секреты GitHub |
| Страница 404 | Неверная конфигурация GitHub Pages | Проверить настройки деплоя |

### Бот не отвечает

| Симптом | Возможная причина | Решение |
|---|---|---|
| Бот молчит | Callback API не настроен | Проверить URL и подтверждение в настройках ВК |
| Бот отвечает ошибкой БД | Проблема подключения к Supabase | Проверить параметры Connection Pooler |
| Бот не узнаёт пользователя | VK ID не в маппинге | Добавить запись в `VK_TO_EMPLOYEE_MAP` |

### Данные не появляются на сайте

| Симптом | Возможная причина | Решение |
|---|---|---|
| Смена добавлена через бота, но не видна | Кэш сайта не обновился | Обновить страницу или подождать Realtime |
| Данные есть в БД, но не отображаются | Ошибка в запросе или фильтрации | Проверить консоль браузера |

## 11. Рекомендации по безопасности

### Общие рекомендации

- никогда не коммитьте файл `.env` в репозиторий;
- добавьте `.env` в файл `.gitignore`;
- используйте секреты GitHub для production-переменных;
- регулярно обновляйте зависимости проекта.

### Ключи и токены

- `VITE_SUPABASE_ANON_KEY` — публичный ключ, но его всё равно не следует размещать в открытых источниках без необходимости;
- `VK_TOKEN` — токен сообщества, должен храниться только на сервере бота;
- пароль базы данных — должен храниться только в конфигурации бота.

### Row Level Security

- рекомендуется включить RLS для всех таблиц с пользовательскими данными;
- настроить политики так, чтобы сотрудники видели только свои записи;
- администраторы должны иметь полный доступ;
- анонимный доступ должен быть закрыт.

### Доступ к базе данных

- используйте Connection Pooler для подключений из serverless-функций;
- ограничьте прямой доступ к базе данных;
- периодически меняйте пароль базы данных;
- следите за количеством активных подключений.
